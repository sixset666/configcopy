Обработка "Обмен с DMSBB"

Назначение

Обработка "Обмен с DMSBB" предназначена для осуществления обмена данными между информационными базами 1С:Предприятие и сервисами, зарегистрированными в системе DMSBB.

В качестве данных сервисов могут выступать сервисы таких приложений как ETKA и ELSA. Обработка позволяет предоставлять информацию о заказ-нарядах, заказах покупателя, автомобилях, контрагентах, запчастях.
Также доступен функционал по регистрации новых сервисов в DMSBB.

Основной задачей DMS-Backbone (DMS-BB) является обеспечение центральной коммуникационной платформы для приложений и информационных сервисов у дилеров и импортеров Volkswagen Group
При этом каждый из сервисов должен знать физический адрес всех остальных сервисов. Кроме того, приложения должны интегрироваться с различными интерфейсами передачи данных для каждого сервиса, результатом чего являются большие временные затраты на разработку и внедрение. DMSBB позволяет решить эти задачи и выступает в роли сервиса-посредника (proxy - сервера)

В случае использования DMSBB, дресация между сервисами происходит по их логическими адресам, а приложения настраиваются на единственный физический адрес – DMSBB. Общая таблица физических адресов хранится в базе данных DMSBB. DMSBB позволяет установить связь сервисов импортера с существующей системой управления дилерским предприятием (1С: Предприятие). Это позволяет оптимизировать бизнес процессы послепродажного обслуживания автомобилей.

 

Сервисы Альфа-Авто

Обработка «Обмен с DMSBB» включается в себя набор стандартных сервисов, позволяющий производить перенос данных в такие приложения как ETKA и ELSA. К таким сервисам относятся:

  • WorkshopOrder - сервис, предоставляющий информацию о заказ-нарядах);
  • SparepartOrder - сервис, предоставляющий информацию о заказах покупателя);
  • ShopData - сервис, предоставляющий информацию о ценах на номенклатуру/запчасти);
  • CustomerVehicleData - сервис, предоставляющий информацию об автомобилях);
  • BusinessPartnerData - сервис, предоставляющий информацию о контрагентах).

Каждый из приведенных выше сервисов имеет свой набор методов. Данные методы отвечают за предоставление и получение соответствующей информации.

WorkshopOrder

  • AliveTest. Метод возвращает список доступных методов существующих сервисов.
  • ListEntries. Метод возвращает список заказ-нарядов в сокращенном варианте. В список попадают только те заказ-наряды, которые не были выполнены/закрыты, т.к. закрытый заказ-наряд не подлежит редактированию.
  • GetEntry. Метод возвращает сокращенную информацию по заданному заказ-наряду.
  • ListOrder. Метод возвращает список незакрытых заказ-нарядов с расширенной информацией.
  • GetOrder. Метод возвращает полную информацию об указанном заказ-наряде.
  • UpdateOrder. Метод осуществляется обновление данных заказ-наряда. Производится редактирование состава табличных частей Товары (Sparepart) и Услуги (Labour_operation).
  • DeleteOrderContent. Метод осуществляет очистку указанных табличных частей заданного заказ-наряда.
  • SubscriptionData. Метод производит подписку заданного сервиса на события WorkshopOrder.
  • SubscriptionData. Метод предназначен для передачи данных об изменениях заказ-наряда соответствующим подписчикам на изменения.

SparepartOrder

  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • ListEntries. Метод возвращает список заказов покупателя в сокращенном варианте. В список попадают только непроведенные заказы, т.к. проведенный заказ покупателя не подлежит редактированию.
  • GetEntry. Метод возвращает сокращенную информацию по заданному заказу покупателя.
  • ListOrder. Метод возвращает список непроведенных заказов покупателя с расширенной информацией.
  • GetOrder. Метод возвращает полную информацию об указанном заказе покупателя.
  • UpdateOrder. Метод осуществляется обновление данных заказа покупателя. Производится редактирование состава табличной части Товары (Sparepart).
  • DeleteOrderContent. Метод осуществляет очистку табличной части заданного заказа покупателя.
  • Subscription. Метод производит подписку заданного сервиса на события SparepartOrder.
  • SubscriptionData. Метод предназначен для передачи данных об изменениях заказов покупателя соответствующим подписчикам на изменения.

ShopData

  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • GetStockInformation. Метод возвращает информацию о цене указанной номенклатуры (запчасти) с указанием количества и местонахождения. Тип цен задается как настройка обмена.

CustomerVehicleData

  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • ListEntries. Метод возвращает список автомобилей.
  • GetEntry. Метод возвращает данные по указанному с помощью VIN-а автомобилю.
  • UpdateEntry. Метод осуществляет обновление информации об автомобиле. Обычно вызывается из ELS-ы при выборе автомобиля заказа и обновляет данные только о номере комплектации, коде двигателя и коде трансмиссии.

BusinessPartnerData

  • AliveTest. Метод возвращем список доступных методов существующих сервисов.
  • ListEntries. Метод возвращает список контрагентов.
  • GetEntry. Метод возвращает данные по указанному контрагенту.
  • UpdateEntry. Метод осуществляет обновление информации о контрагенте.

Прием запросов от внешних сервисов DMSBB осуществляется при помощи микросервера, который прослушивает заданный tcp-порт и посредством COM-соединения с Альфа-Авто передает запрос и получает на него ответ для обратной передачи. Перед осуществлением обмена данными между Альфа-Авто и DMSBB следует убедиться, что микросервер запушен и подключен к корректной базе 1С.

Относится к подсистеме: «Сервис».
